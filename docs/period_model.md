# Period Recording Model

**日期：** 2026-03-16

## 结论

月经记录模块采用三层模型：

1. 用户层：`Day selection model`
2. 系统层：`Day state as source of truth`
3. 推导层：`Cycle as derived object`

一句话总结：

> 交互上按“区间选择”来录，数据上按“日状态”来存，产品上按“系统推导周期”来展示。

## 目标

让用户以最低心智负担完成月经记录，同时保证系统仍能稳定支持：

- 当前状态展示
- 周期识别
- 下次预测
- 历史回顾
- 删除、补选、拆分等编辑场景

## 核心判断

### 用户不直接创建周期

用户不需要手动创建 `cycle`，也不需要先理解“开始日 / 结束日对象”。

用户真正做的事情只有两类：

- 在时间带上长按并选择一段日期
- 轻点某一天并编辑当天状态

### 周期不是输入对象，而是派生对象

系统根据连续的日状态自动识别经期区块，并在 UI 中展示为周期。

因此：

- 用户编辑的是 day
- 系统维护的是 cycle
- 页面展示的是连续 block

## v1 记录模型

### 原子对象：Day Record

`day_record` 是系统唯一真相，表示某个 profile 在某一天的显式记录。

建议字段：

- `date`
- `profile_id`
- `bleeding_state`
- `flow_level`
- `pain_level`
- `color`
- `notes`
- `source`
- `updated_at`

### 日状态定义

v1 主状态只保留：

- `none`
- `period`
- `spotting`

其中：

- `none` 是产品语义上的合法状态
- v1 存储层不需要为 `none` 单独落库
- 没有 `day_record` 的日期，UI 默认解释为 `none`

也就是说：

- 显式记录：`period` / `spotting`
- 缺省空白：等价于 `none`

## v1 交互模型

### 默认态

- 首页主视图为 `Cycle Window（三周窗口）`
- 轻点某天：查看当天详情并做单日编辑
- 不进入选择态

### 多选态

- 长按某天：进入多选态
- 当前日期成为锚点
- 拖动选择连续日期范围
- 点击“保存”后批量写入 day records
- 点击空白处或退出按钮，离开多选态

### 多选保存的默认行为

多选态保存后，选中的日期默认写入：

- `bleeding_state = period`
- 其他细节字段为空

这表示“我确认这几天属于正常经期记录”，而不是要求用户逐天填细节。

### 单日编辑

轻点某天后，允许对当天做最小编辑：

- 设为 `period`
- 设为 `spotting`
- 清除显式记录
- 补充 `flow_level / pain_level / notes`

这里不单独定义“是否出血”布尔值，避免与主状态冲突。

## 周期推导规则

### Rule 1：连续 `period` 自动合并为一个周期

连续若干天的 `bleeding_state = period` 会被系统识别为一个周期区块。

### Rule 2：`spotting` 不参与 v1 周期边界

`spotting` 只作为日状态展示，不自动并入周期主体，也不决定开始/结束边界。

### Rule 3：删除中间某一天会触发周期重算

如果连续区块中间某一天被清除，系统会把它拆成两个区块。

### Rule 4：补选缺失日期会触发自动合并

如果两段 `period` 之间补上缺失日期，系统自动合并为一个区块。

### Rule 5：允许单日周期存在

v1 不强制周期至少持续两天，单日 `period` 也是合法记录。

## 预测逻辑

预测应基于系统识别出的 `cycle start dates`，而不是用户显式维护的结束日。

v1 规则：

- 从最近 N 次周期的 `start_date` 计算周期长度
- 用平均值或中位数生成下一次预测开始日
- 用 `predicted_start ± 2 days` 形成预测窗口

`end_date` 仅用于展示和统计持续天数，不作为预测核心输入。

## 首页视图边界

### 主视图

首页默认展示：

- 状态卡
- 预测摘要
- 3 周连续时间带
- 单日详情面板

### 月历视图

Month View 只作为辅助浏览和兼容旧习惯的入口，不承担主录入职责。

也就是说：

- 高频记录发生在首页时间带
- 月历主要用于回看和定位历史日期

## v1 必须冻结的规则

- 原子单位是 `day_record`
- 未记录日期按 `none` 解释
- 首页主视图是 3 周连续时间带
- 长按进入多选态，拖选后保存
- 轻点进入单日详情与编辑
- 连续 `period` 自动派生为周期区块
- `spotting` 不参与 v1 周期边界
- 预测基于 `cycle start`

## v1 例子

### 删除中间某一天

- 已有 `2026-03-14 ~ 2026-03-16` 的连续 `period`
- 清除 `2026-03-15`
- 系统应得到两个派生区块：`2026-03-14` 和 `2026-03-16`

### 补齐缺失日并合并

- 已有 `2026-03-14` 与 `2026-03-16` 两个单独 `period`
- 将 `2026-03-15` 设为 `period`
- 系统应自动合并为一个 `2026-03-14 ~ 2026-03-16` 区块

### 单日周期

- 仅 `2026-03-14` 为 `period`
- 系统应保留单日周期，不强制扩展

### `spotting` 邻接经期

- `2026-03-13 = spotting`
- `2026-03-14 ~ 2026-03-15 = period`
- `2026-03-16 = spotting`
- 派生区块边界仍然只覆盖 `2026-03-14 ~ 2026-03-15`

## v1 明确不做

- `spotting` 并入周期边界的高级规则
- 缺失 1 天自动补全连续性的 tolerance 规则
- AI 异常识别
- 高级统计与复杂健康分析

## 对项目文档的影响

这套模型落地后，相关设计与实现文档都应统一为：

- `day_record` 是真相
- `cycle` 是系统派生对象或缓存对象
- “补录”不是独立心智模型，而是“编辑今天及之前任意日期”
- “今天来了 / 今天结束了”只能作为快捷入口，不能成为底层数据模型本身
